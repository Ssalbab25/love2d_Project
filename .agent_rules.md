# 목적

앞으로 작성하는 모든 Lua 및 Love2D 코드는 다음의 SOLID 원칙을 엄격하게 준수해야 합니다. 
Lua는 클래스나 인터페이스가 내장되어 있지 않으므로 메타테이블(Metatable)과 클로저(Closure), Duck Typing을 
활용하여 아래의 원칙을 논리적으로 강제하십시오.

## 절차

1. SRP (단일 책임 원칙 - Single Responsibility Principle)

지침: Love2D의 main.lua는 애플리케이션의 진입점 역할만 수행해야 합니다. 게임 로직, 렌더링, 입력 처리를 절대 한 파일이나 하나의 테이블에 섞지 마십시오.
체스 적용: * Board.lua: 체스판의 데이터(2차원 배열) 상태만 관리합니다.
MoveValidator.lua: 기물의 이동 가능 여부와 체크/체크메이트 판별 등 규칙만 계산합니다.
ChessRenderer.lua: Love2D의 love.graphics API를 호출하여 화면에 그리는 역할만 담당합니다.

2. OCP (개방-폐쇄 원칙 - Open/Closed Principle)

지침: 기존 코드를 수정하지 않고도 새로운 기능을 추가할 수 있어야 합니다. 조건문(if type == "knight" then...)으로 기물의 이동을 분기 처리하는 것을 엄격히 금지합니다.
체스 적용: 기본이 되는 Piece 모듈을 만들고, Pawn, Knight, Queen 등은 이를 상속(또는 확장)하여 각자 고유한 getValidMoves() 함수를 구현해야 합니다.
Board는 기물의 종류를 알 필요 없이 단순히 선택된 기물의 getValidMoves()를 호출하기만 해야 합니다.

3. LSP (리스코프 치환 원칙 - Liskov Substitution Principle)

지침: 자식 객체는 언제나 부모 객체를 대체할 수 있어야 합니다. Lua에서는 명시적인 타입 체크가 없으므로 반환값의 형태를 엄격하게 통일하십시오.
체스 적용: * 모든 체스 기물 객체는 getValidMoves(boardState, currentPos)를 호출했을 때, 반드시 { {x, y}, {x, y} ... } 형태의 동일한 테이블 구조를 반환해야 합니다. 어떤 기물은 nil을 반환하고 어떤 기물은 에러를 던지는 식의 일관성 없는 행동을 해서는 안 됩니다.

4. ISP (인터페이스 분리 원칙 - Interface Segregation Principle)

지침: 클라이언트(호출자)가 자신이 사용하지 않는 메서드에 의존하지 않도록, 거대한 범용 테이블(God Object) 생성을 피하십시오.
체스 적용:
입력 처리(Input Handling)를 분리하십시오. 마우스 클릭을 처리하는 로직과 키보드 단축키(예: 무르기 기능)를 처리하는 로직을 분리하여 필요한 모듈에만 연결(require)되도록 구성하십시오.

5. DIP (의존성 역전 원칙 - Dependency Inversion Principle)

지침: 고수준 모듈(게임 규칙)이 저수준 모듈(Love2D 그래픽 API)에 직접 의존해서는 안 됩니다.
체스 적용: 게임 로직 모델(Board, Piece) 내부에서 love.graphics나 love.audio를 직접 호출하지 마십시오.
게임 로직은 순수한 Lua 코드로만 작성되어야 하며, Renderer나 SoundManager가 게임 모델의 상태를 읽어가서 Love2D API를 호출하도록 의존성을 분리(의존성 주입)하십시오.

## 코드 작성 규칙

글로벌 변수(_G) 사용을 엄격히 금지합니다. 모든 모듈은 local로 선언하고 명시적으로 return 하십시오.

복잡한 로직 작성 시, 코드 작성 전 주석으로 이 코드가 SOLID의 어떤 원칙을 어떻게 만족하는지 짧게 명시하십시오.
