# [게임알고리즘(3A)] 기말과제 안내 (바이브 코딩 프로젝트)

안녕하세요, 여러분.
게임알고리즘(3A)에서는 게임최적화기법(2A)과 **같은 프로젝트**를 제출하되,
평가 포인트는 "결과물의 완성도와 실무 구현 품질"에 맞춰 진행합니다.

핵심만 먼저 안내할게요.
- 프로젝트는 **1개**입니다.
- 제출은 **게임알고리즘(3A) 과제 게시판**에 따로 합니다.
- 이 과목은 "잘 동작하고, 설명 가능하고, 재현 가능한 결과물"을 봅니다.

---

## 1. 과제 개요
- 과제 형태: 개인 또는 팀 프로젝트 (수업 운영 기준에 따름)
- 팀 제출 기준: 팀당 최대 (직접 입력)명
- 팀 제출 시 개인 기여 증빙 필수: 커밋 이력 + 역할/작업 분담표
- 개발 방식: AI 협업 가능 (권장)
- 목표: 요구사항을 충족하는 안정적인 소프트웨어 결과물 제출

---

## 2. 점수 기준 (30점 만점)
1. 요구사항 충족도: 10점
2. 코드 구조와 품질: 8점
3. 안정성 및 테스트 결과: 6점
4. 결과물 전달력(문서/데모/설명): 6점

### 쉽게 말하면
- "아이디어"보다 "동작하는 결과"를 우선 평가합니다.
- 실행 가능성, 재현성, 설명 가능성이 중요합니다.

---

## 3. 제출 방법 (게임알고리즘(3A) 게시판)
- 제출 위치: **게임알고리즘(3A) 과제 게시판**
- 제출 형식: 압축파일 1개 + 보고서(`.md` 권장, `.txt` 허용) + 데모 영상 링크
- 권장 파일명: `학번_이름_게임알고리즘3A_기말.zip`
- 동일 프로젝트 제출 정책: 같은 프로젝트 산출물(코드/로그/테스트)은 과목 간 재사용 가능
- 단, 과목별 보고서 파일은 과목명에 맞게 구분해 제출

---

## 4. 제출 구조 규칙 (수업 공통 템플릿)
제출물은 수업 프로젝트와 **동일한 기본 구조**로 정리해 주세요.

필수 구조(이름/위치 유지)
- `project/src/` : 실제 실행 코드
- `project/tests/` : 테스트/검증 코드 또는 하네스
- `docs/` : 설계/요약 문서
- `worklogs/` : 작업 로그 또는 회고 기록
- `run.sh` 또는 `run.bat` : 실행 진입 스크립트(가능한 경우)
- 루트 `README.md` : 설치/실행/검증 방법

선택 구조(자유 확장)
- `assets/`, `tools/`, `scripts/`, `experiments/` 등은 자유롭게 추가 가능

구조 관련 운영 원칙
- 필수 구조가 일부 누락되면 즉시 0점 처리하지 않고, 안내 후 재제출 1회를 허용합니다.
- 재제출 이후에도 필수 구조가 불명확하면 감점될 수 있습니다.

## 7. 자주 묻는 질문
Q1. 게임최적화기법(2A)와 게임알고리즘(3A)을 따로 개발해야 하나요?
- 아니요. **하나의 프로젝트**를 두 과목 게시판에 각각 제출하면 됩니다.

Q2. 두 게시판에 완전히 다른 파일을 내야 하나요?
- 아닙니다. 공통 산출물은 같아도 됩니다.
- 대신 게임알고리즘(3A)에는 이 과목 관점 보고서와 데모를 꼭 포함하세요.

Q3. 실행이 되지만 일부 버그가 있으면 0점인가요?
- 0점은 아닙니다.
- 다만 버그의 영향, 원인, 개선 계획이 제출물에 명확해야 합니다.
