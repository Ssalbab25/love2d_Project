# AI 사용 가이드

이 문서는 이 저장소에서 AI를 효율적으로 쓰기 위한 기준을 정리합니다. 목표는 토큰을 아끼면서도, Copilot / Windsurf / Antigravity에서 같은 판단 기준으로 에이전트와 스킬을 고르는 것입니다.

---

## 1. 토큰을 아끼는 기본 원칙

- 한 번에 하나의 목표만 요청한다.
- 파일명, 에러 메시지, 재현 방법처럼 근거가 되는 정보를 먼저 준다.
- “좋게 해줘” 대신 “무엇을, 어디서, 어떤 기준으로”를 적는다.
- 넓은 탐색보다 바로 바꿀 파일을 먼저 지목한다.
- 동일한 배경 설명은 매번 반복하지 말고 문서를 참조하게 한다.
- 필요한 경우에만 깊은 설명을 요청하고, 기본은 짧은 답변을 선호한다.

### 검증 우선순위 (중요)

- 기본 검증은 테스트 하네스(단위/모듈/시뮬레이션) 우선이다.
- LÖVE2D 전체 실행 검증은 최소한으로 수행한다.
- 전체 실행은 입력/렌더/프레임 타이밍처럼 하네스로 확인하기 어려운 변경에서만 1회 이상 수행한다.
- 요청 시 AI는 "하네스 결과"와 "최소 실행 검증 결과"를 분리해서 보고한다.

### 좋은 요청의 형태

- 목표
- 대상 파일 또는 범위
- 제약 조건
- 원하는 출력 형식
- 검증 방법

예시:
- `project/src/main.lua에서 입력 처리만 분리해줘. update/draw 분리는 유지하고, 검증은 Lua: Run current file로 해줘.`

## 2. 도구별 역할 분담

### Copilot

- `prompt-expert`로 요청을 먼저 다듬는다.
- `lua-expert`는 Lua 문법, 테이블, 메타테이블, 성능, 오류 처리에 사용한다.
- `love2d-architect`는 구조, 의존성, 모듈 경계에 사용한다.
- `love2d-feature-scaffold`는 기능 추가, 상태 분기, update/draw 분리에 사용한다.
- `release-check`는 변경 후 품질 점검에 사용한다.

### Windsurf

- `rules`는 항상 필요한 제약을 담는다.
- `skills`는 반복되는 다단계 작업에 사용한다.
- `workflows`는 `/workflow-name`처럼 수동 실행형 절차에 사용한다.
- 문서가 길어질수록 “무엇이 자동 적용되는지”와 “무엇을 직접 호출해야 하는지”를 분리한다.

### Antigravity

- `rules`는 핵심 코딩 기준을 담는다.
- `skills`는 구조 개선, 기능 스캐폴딩, 점검 같은 작업에 사용한다.
- `workflows`는 점검, 릴리스, 프롬프트 정리처럼 반복되는 절차에 사용한다.
- Windows 예시가 있더라도, 공통 안내는 task 기반으로 먼저 설명한다.

## 3. 환경별 추천 사용법

### 공통

1. 먼저 목적을 한 문장으로 정리한다.
2. 해당하는 파일이나 디렉토리를 지정한다.
3. 적용할 도구를 고른다.
4. 검증 방법을 함께 적는다. (하네스 우선, 전체 실행은 필요 시)
5. 결과가 길어질 것 같으면 “짧게”와 “상세하게” 중 하나를 먼저 정한다.

### macOS / Linux 학생

- 실행은 `run.sh` 또는 `Love2D: Run project` task를 우선한다.
- Lua 단위 확인은 `Lua: Run current file`을 우선한다.
- 브라우저, 터미널, 에디터가 섞여 있어도 task 이름은 공통으로 유지한다.

### Windows 학생

- 실행은 `run.bat` 또는 `Love2D: Run project` task를 우선한다.
- 로컬 런타임이 필요한 경우 저장소 포함 바이너리를 보조 경로로 쓴다.
- F5는 가능하면 보조 경로로만 쓰고, task를 기본 경로로 둔다.

## 4. 어떤 도구를 먼저 쓸지

### 먼저 `prompt-expert`

- 요청이 길고 막연할 때
- 여러 제약이 섞여 있을 때
- AI에게 무엇을 물어야 할지 정리할 때

### 바로 `lua-expert`

- Lua 문법, 성능, 오류 처리, 테이블 구조가 핵심일 때
- 코드가 Lua 레벨에서 왜 깨지는지 볼 때

### 바로 `love2d-architect`

- 모듈이 커졌을 때
- update/draw 경계가 흐려졌을 때
- 의존성을 줄여야 할 때

### 바로 `love2d-feature-scaffold`

- 새 기능을 추가할 때
- 기존 기능을 최소 변경으로 나눌 때
- 상태 전이와 렌더링을 분리할 때

### 마지막에 `release-check`

- 수정 후 전체 리스크를 빠르게 보고 싶을 때
- 배포 직전 체크가 필요할 때

---

## 5. 추천 작업 흐름

1. 사용자가 할 일을 `prompt-expert`로 정리한다.
2. Lua 자체 문제면 `lua-expert`로 깊게 들어간다.
3. 구조 문제면 `love2d-architect`로 재설계한다.
4. 기능 추가면 `love2d-feature-scaffold`로 구현한다.
5. 마무리로 `release-check`를 돌린다.

## 6. 실전 예시

### 예시 1: 입력 처리 정리

- 목표: 입력 처리와 렌더링 분리
- 도구: `love2d-feature-scaffold`
- 검증: 하네스 우선 + 필요 시 `Love2D: Run project` 1회

### 예시 2: Lua 모듈 에러 수정

- 목표: nil 에러 해결
- 도구: `lua-expert`
- 검증: `Lua: Run current file`

### 예시 3: 복잡한 요청을 짧게 바꾸기

- 목표: AI가 바로 수행 가능한 형태로 요청 재작성
- 도구: `prompt-expert`
